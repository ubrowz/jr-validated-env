Get Started

Set up the JR Validated Environment on your team

Requirements

  • macOS (Apple Silicon or Intel)
  • R — version specified in admin/r_version.txt
  • Python — version specified in admin/python_version.txt
  • Dropbox — for sharing the local package repository across the team
  • Xcode Command Line Tools — run xcode-select --install in Terminal if not already installed

Package versions are pinned in requirements files. The environment installs from a local Dropbox repository — no internet access required once the admin has built the repo.

Quick start — End users
1

Make sure Dropbox is installed and fully synced on your Mac before continuing.

2

Open Terminal: press ⌘ Space, type Terminal, press Enter.

3

Find setup_jr_path.zsh in the JR project folder in Finder. Drag it into the Terminal window and press Enter.

4

You will see:

✅ PATH updated successfully.
5

Open a new Terminal window (⌘ N). You are ready.

6

Type the name of any JR script and press Enter. On first run the environment builds automatically — this may take a minute. All subsequent runs are fast.

You only need to run setup_jr_path.zsh once per machine. Use the Script Guide to find the right script for your analysis.
Check installed versions

To see the currently installed R, Python, and package versions at any time:

jr_versions
First-time setup

Requires internet access to download packages.

1

Clone the repository:

git clone https://github.com/ubrowz/jr-validated-env.git cd jr-validated-env
2

Build the local package repositories and install the environments:

./admin/admin_install_R --rebuild ./admin/admin_install_Python --rebuild
3

Generate the project integrity file:

./admin/admin_create_hash
4

Generate validation scripts and confirm the environment is working:

./admin/admin_validate

A timestamped IQ evidence file is written to ~/.jrscript/<PROJECT_ID>/validation/ — suitable for an audit or FDA submission.

Subsequent setups

No internet needed — installs from the local Dropbox repository.

./admin/admin_install_R ./admin/admin_install_Python
Adding new team members

Share the JR project folder via Dropbox (already synced for existing members). The new team member runs setup_jr_path.zsh once on their machine — the environment is built automatically on first script run.

For teams with a mix of Apple Silicon and Intel Macs, see the multi-architecture section in docs/admin_manual.pdf.

OQ validation

To run the full OQ test suite against all community scripts:

1

Pre-flight check — verifies the OQ environment before running:

./admin/admin_oq_validate
2

Run the OQ test suite and produce a timestamped evidence file:

./admin/admin_oq

Evidence is written to ~/.jrscript/<PROJECT_ID>/validation/oq_execution_<timestamp>.txt.

Adapting for your project

There are two ways to use the JR Validated Environment depending on your needs.

Option 1 — Install and configure (recommended)

Download from GitHub Releases and follow the Admin Manual. After installation, the admin configures the environment for the project:

  1. Edit admin/R_requirements.txt and admin/python_requirements.txt with the packages your scripts require
  2. Edit admin/r_version.txt and admin/python_version.txt with the R and Python versions to pin
  3. Run ./admin/admin_install_R --rebuild and ./admin/admin_install_Python --rebuild to build the local package repository
  4. Add R and Python scripts to the R/ and Python/ subfolders following the Admin Manual
  5. Run ./admin/admin_create_hash then ./admin/admin_validate to generate validation artefacts

Team members then run setup_jr_path.zsh once on their machine. The environment is ready.

Option 2 — Fork and extend

To modify the architecture, contribute improvements, or significantly extend the framework for your own purposes, fork the repository on GitHub and make your changes. Please open an issue before submitting a pull request so the proposed change can be discussed first.

github.com/ubrowz/jr-validated-env

The validation evidence included in docs/ covers the specific R, Python, and package versions at the time of release. If you install with different versions — because newer installers are available, or because you add or update packages — the included evidence no longer applies. You must perform your own revalidation using the templates in docs/templates/ before using the environment in a regulated context.
