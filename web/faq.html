Frequently Asked Questions

Answers for analysts, administrators, and quality managers

Using the scripts
Do I need to know R or Python to use the scripts?

No. You type the script name in Terminal and press Enter — the environment handles everything else. You will need to prepare a CSV data file in the correct format, but no programming knowledge is required to run an analysis.

What format does my data need to be in?

All analysis scripts expect a two-column CSV file with the headers id and value. The id column is a sample or unit identifier; the value column holds the measurement.

If your data is in a different format — multi-column, tab-delimited, or with header rows to skip — use jrc_convert_csv or jrc_convert_txt to convert it first. Run either script with --help for options.

Where do output files (plots, result CSVs) appear?

Output files are written to the current working directory — wherever your Terminal is pointed when you run the script. To control where outputs land, cd to your preferred folder before running the script, or move files afterwards.

My first run took a long time — is that normal?

Yes. On the very first run on a new machine, the environment builds automatically: R packages are installed from the local Dropbox repository and the Python venv is created. This is a one-time step and typically takes 1–3 minutes depending on the number of packages. All subsequent runs start in a few seconds.

If the build fails, confirm that Dropbox is installed, signed in, and fully synced before retrying.

Can I run a script directly in RStudio, Jupyter, or a plain terminal with Rscript?

No. Every R script checks at startup that it was launched through the jrrun wrapper. If that check fails the script halts immediately with a clear error message. This is intentional — the wrapper sets the controlled package library and logs the run. Running outside the wrapper would bypass both controls.

If you need interactive R analysis for exploratory work, keep that separate from validated analysis runs. All validation evidence must come from wrapper-executed runs.

Will I get the same result as a colleague on a different Mac?

Yes, provided both machines have installed the environment from the same Dropbox repository. Every team member runs exactly the same R and Python versions with exactly the same package versions — the environment is identical across machines. The integrity check confirms that the project files themselves have not changed.

I got an error — where do I start diagnosing it?

The most common issues are covered in docs/TROUBLESHOOTING.md, which is included in the project folder. The first things to check:

1. Is Dropbox running and fully synced?
2. Did you open a new Terminal window after running setup_jr_path.zsh?
3. Is the error an integrity check failure? If so, a project file has changed since the hash was generated — contact your administrator.
4. Run jr_versions to confirm R, Python, and all packages are installed correctly.

A run log at ~/.jrscript/<PROJECT_ID>/run.log records every execution with timestamp, arguments, and exit code — useful for tracing exactly what ran and when.

Setup and administration
Why not Docker?

Docker solves a real problem — reproducible environments — but it is aimed at software engineers. The JR Validated Environment is aimed at analysts and quality professionals who need controlled, auditable statistical analysis without DevOps overhead.

JR Validated Environment Docker
Learning curve Low — basic Terminal only High — images, registries, Dockerfile
Audit transparency High — plain text requirements files Moderate — binary image requires tooling
Resource usage Minimal — no background processes Heavy — Linux VM always running
Package updates Edit one file, auto-propagated Rebuild and redistribute entire image

Choose the JR Validated Environment if your team is macOS-based, consists of analysts rather than software engineers, and you want validation evidence in plain text files that a Quality Manager can read directly. See the README for the full comparison including offline use, cross-platform considerations, and more.

How can I arrange shared folders for my team?

The environment needs the package repository accessible as a folder path on each machine. Two options work well for most teams:

  • Dropbox™ Syncs the package repository automatically to each team member's machine, works offline, and requires no IT infrastructure. Dropbox is the path described in the Admin Manual and the one used during validation — it is the recommended choice for most teams.
  • SMB file sharing (built into macOS) The administrator enables File Sharing in System Settings; the team mounts the folder via Finder (⌘ K). No extra software is needed. Works well with a dedicated machine — a Mac mini or a NAS device such as a Synology — acting as the file server. Note that SMB is live-mounting, not syncing: the host machine must be on and reachable, and there is no offline access if the connection drops. Pairing SMB with Git on the host is recommended to provide version history and a full audit trail of changes.

If you use a mechanism other than Dropbox, confirm it presents a consistent, readable folder path before relying on it in a regulated context.

Does the environment need internet access at runtime?

No. Once the administrator has built the local package repository (which does require internet access the first time), all subsequent installs — including new team member setups — run entirely from the local Dropbox copy. The validated environment never fetches packages from the internet at runtime.

Can I use this on Windows or Linux?

No — macOS only (Apple Silicon and Intel are both supported). The environment relies on macOS-specific paths and zsh shell behaviour. The package repository may also contain platform-specific binaries. If your team uses Windows machines, the JR Validated Environment is not currently an option.

How do I add a new team member?

Share the JR project folder via Dropbox — if it is already in a shared Dropbox folder, simply invite the new member to that folder. They then:

1. Install the correct R and Python versions (installers are in the Dropbox repository).
2. Wait for Dropbox to finish syncing.
3. Run setup_jr_path.zsh once from the project folder to add the environment to their PATH.
4. Open a new Terminal window — the environment is ready. It will build automatically on their first script run.

No further admin action is required for each new user. For teams with a mix of Apple Silicon and Intel Macs, see the multi-architecture section in docs/admin_manual.pdf.

How do I add a new R or Python package?

Use the --add flag on the relevant install script. This downloads only the new package (and its dependencies) without rebuilding the entire repository:

./admin/admin_install_R --add pkgname==1.2.3
./admin/admin_install_Python --add pkgname==1.2.3

After adding a package you must re-run ./admin/admin_create_hash to update the integrity file, and then re-run ./admin/admin_validate to generate fresh IQ evidence. Adding a package is a change to the validated configuration and requires revalidation before using affected scripts in a regulated context.

The integrity check is failing after I updated a script — is that expected?

Yes — this is the system working as intended. Any change to a tracked project file causes the SHA256 integrity check to fail until the integrity file is regenerated.

Once you have reviewed and approved the change, run ./admin/admin_create_hash to update the integrity file. If the change was unintentional, restore the file from git first (git checkout <filename>) and do not regenerate the hash. Regenerating the hash without reviewing changes is a Quality event — treat it accordingly.

What is the difference between admin_validate and admin_oq?

admin_validate performs Installation Qualification (IQ): it verifies that R, Python, and all pinned packages are correctly installed and writes a timestamped evidence file. Run this when setting up a new machine or after any configuration change.

admin_oq runs the automated Operational Qualification (OQ) test suite: 142 pytest-based tests that exercise every community script with known inputs and check the outputs. Run this to generate OQ evidence for an audit or submission, or after any change that could affect script behaviour.

Both produce timestamped evidence files in ~/.jrscript/<PROJECT_ID>/validation/.

Validation and compliance
Does the included validation evidence cover my installation?

Only if you install with exactly the same R, Python, and package versions that were used when the included documents were produced. If you use different versions — because newer installers are available, or because you add or update packages — the included evidence no longer applies to your configuration.

In that case you must perform your own revalidation. Validation plan and report templates are in docs/templates/. The admin_validate and admin_oq commands generate the evidence files you need to complete them.

What changes trigger a revalidation?

Any of the following require re-running admin_create_hash and potentially producing new validation evidence:

— Adding, removing, or changing the version of any R or Python package
— Changing the pinned R or Python version
— Modifying any script, wrapper, or admin file covered by the integrity check
— Adding a new community script

Pure documentation changes (help text, comments) that do not alter script behaviour may not require a full re-OQ, but they do require a new integrity hash. Your Quality Manager should make the final determination based on the nature of the change and your organisation's change control procedures.

What do I show an auditor?

The primary evidence artefacts are:

— IQ evidence — generated by admin_validate; confirms correct installation on the specific machine, with timestamp.
— OQ evidence — generated by admin_oq; 142 automated test results with pass/fail and timestamp.
— Validation plan (docs/core_IQ_validation_plan.pdf, docs/statistics_validation_plan.pdf) — documents the test approach and acceptance criteria.
— Validation report (docs/core_validation_report.pdf, docs/statistics_validation_report.pdf) — summarises test execution and results.
— Run log at ~/.jrscript/<PROJECT_ID>/run.log — audit trail of every script execution.

All evidence files are plain text or PDF — no special tooling required to read them.

Is there a cost to use this?

The JR Validated Environment itself is free and open source (Apache 2.0). You provide:

— macOS machines for your team
— Dropbox (or equivalent shared storage) to distribute the package repository
— The correct R and Python installers, available from their respective open-source projects

There is no subscription, licence fee, or per-user charge for the environment itself.

Who is responsible for the statistical correctness of the scripts?

The validation evidence covers Operational Qualification — it confirms that each script produces the correct numerical output for a set of known inputs. It does not certify that a given script is the right choice for every possible study design or application.

The engineer or quality professional applying the environment to a specific study is responsible for verifying that the selected script and its assumptions are appropriate for their intended use. The Script Guide and the per-script help text (jrrun <script> --help) describe the applicable conditions for each script.

Can I use this for FDA submissions (21 CFR Part 11)?

The environment is designed to support 21 CFR Part 11 and ISO 13485 requirements: it provides audit trails (run log), access controls (wrapper-only execution, bypass protection), integrity verification (SHA256 on every run), and documented validation evidence. Whether it meets your specific submission requirements depends on your organisation's validation programme and your FDA reviewer's expectations.

The included validation documents follow GAMP 5 guidance. They serve as a starting point — your Quality Manager should review and augment them to reflect your organisation's procedures and the specific intended use.
